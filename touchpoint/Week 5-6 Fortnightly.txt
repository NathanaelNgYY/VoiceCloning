Week 5 – Sprint 5 Contributions

During Week 5, my main contribution was helping the project move from the earlier lecturer voice-cloning use case into a more interactive speech-based system. The original idea was for lecturers to clone their voice so students could receive online lecture content in a familiar lecturer voice. After that, the use case evolved into a speech-to-speech flow, where a user speaks into the system and receives the same spoken content back as cloned voice output using another trained voice. After this speech-to-speech direction worked successfully, the project was then further integrated into a chatbot experience, where the user can speak to an AI assistant and receive chatbot responses in a cloned voice. This required major changes across the frontend, backend, GPU worker, live gateway, Lambda backend, AWS routing, security setup, and deployment architecture. 

One of my first major tasks was integrating the chatbot into the website. The Live feature was changed from a browser-side recording/transcription flow into a backend-bridged OpenAI Realtime chatbot. The browser now streams microphone audio through a WebSocket connection, the backend/live service connects to OpenAI Realtime, and the assistant reply is converted into cloned voice audio using GPT-SoVITS. This made the system feel more like a real voice assistant instead of just a recording and playback tool. On the backend, I contributed to adding the OpenAI Realtime bridge, including WebSocket handling, browser PCM audio streaming, OpenAI Realtime event handling, app-safe event mapping, backend-only OpenAI API key usage, input transcription for user bubbles, improved production WebSocket origin handling, and fixes such as removing outdated headers and unsupported session parameters. On the frontend, I helped refactor the Live page into a chatbot transcript UI with user and assistant message bubbles, microphone streaming, WebSocket state management, cloned audio playback, interruption handling, and conversation state management.

I also helped create and improve the two chatbot playback modes: **Live Full** and **Live Fast**. Live Full sends the completed assistant reply through the normal inference route, where GPT-SoVITS handles the long-text splitting internally and returns one full WAV file. Live Fast splits the assistant response into smaller punctuation-based phrases, sends each phrase for TTS generation, and queues playback so the response can start faster. I helped fix Live Fast issues such as old audio replaying after interruption, phrase playback stopping early, interruption being allowed before audio was ready, mic-off getting stuck in transcription, and playback not continuing properly after generated clips were already played. I also helped separate controls more clearly, such as microphone control, stopping voice playback, and ending the conversation.

At the same time, I helped remove the old Live-only Faster Whisper path because OpenAI Realtime became the new live chatbot path. This included removing old live upload, live transcription, and phrase transcription routes from the backend, GPU worker, and frontend API helpers. However, the normal training pipeline and standard transcription path were kept intact. The important change was that only the old Live-only ASR path was removed, while training, reference-audio transcription, and normal inference-related functionality remained.

A very large part of Week 5 was migrating the backend from EC2 to Lambda. My supervisor wanted the backend EC2 server removed because Lambda would be cheaper and easier to manage for stateless API work. I helped restructure the system so Lambda handles normal REST API requests, while the GPU server still handles GPU-heavy and long-running work such as training, inference, GPT-SoVITS, SSE progress streams, and live WebSocket communication. This created a clearer architecture: Lambda handles config, uploads, training start/status, inference start/status, model listing, training audio, and live TTS sentence requests; the GPU worker handles training and inference; the live gateway handles the long-lived chatbot WebSocket; S3 acts as shared storage; and CloudFront plus ALB routes traffic to the correct service.

For the Lambda backend, I helped create route modules for config, upload, training, models, inference, transcription, training audio, and live TTS sentence generation. I also helped add shared Lambda utilities for CORS responses, JSON parsing, S3 operations, safe path handling, GPU worker HTTP calls, and artifact URL selection. Upload design was changed so Lambda does not receive large audio files directly. Instead, Lambda creates presigned S3 upload URLs, the frontend uploads files directly to S3, and then the frontend confirms the upload. For training and inference, Lambda only starts or checks jobs, while the GPU worker performs the actual long-running work and streams progress through SSE. I also helped add current training and current inference state support so the frontend can restore progress after refresh.

The Lambda deployment approach also changed during Week 5. The first migration plan used SAM and API Gateway, but the supervisor later requested Lambda Function URL instead. I helped update the code so the Lambda backend can run as a normal Node.js Lambda function using `index.handler`. Since a Function URL gives one HTTP endpoint, I helped add an internal router that checks the request method and path, then sends the request to the correct route module. This allowed one Lambda function to support routes such as config, models, training state, inference state, and live TTS sentence generation. I also helped remove the SAM deployment structure, add a zip packaging workflow, and update the deployment guide so the backend can be packaged and uploaded directly to AWS Lambda.

I also contributed to AWS account and CLI setup. I configured AWS CLI access and role-based account usage so deployment and permission changes could be done from the terminal instead of relying only on the AWS web console. This was important because some IAM and Lambda permission changes, especially for Function URL and CloudFront access, are easier or necessary through AWS CLI. I verified the active AWS identity using STS and ensured deployment actions were targeting the correct AWS account.

Another major contribution was securing the Lambda Function URL behind CloudFront. Since a raw Lambda Function URL can be public, I helped configure it to use AWS IAM authorization and CloudFront Origin Access Control. This means the browser calls CloudFront, CloudFront signs the request to Lambda, and the raw Function URL rejects unsigned public requests. I also helped add the required Lambda permissions so CloudFront could invoke the Function URL correctly. During debugging, I identified issues such as CloudFront returning React HTML instead of backend JSON because a 403 error was being rewritten to `index.html`. I helped correct the SPA fallback approach so 404 errors can still go to the frontend app, but 403 errors remain visible for debugging security and permission problems.

I also worked on CloudFront, ALB, and security group routing. Normal `/api/*` REST requests are routed through CloudFront to the Lambda Function URL. Live WebSocket traffic, training progress SSE, and inference progress SSE are routed through CloudFront to the GPU ALB because those flows require long-running connections. I helped define ALB listener rules and target groups, including one target group for the GPU worker and another for the live gateway. I also helped set safer security group rules so the GPU worker and live gateway only accept traffic from the ALB security group, while the internal GPT-SoVITS API remains unavailable publicly.

To reduce GPU cost, I helped add GPU instance controls and idle shutdown support. The frontend can now check the GPU EC2 instance status and start it from the UI when needed. Lambda supports instance status and start endpoints, allowing users to start the GPU only when required. I also helped implement idle shutdown, where Lambda checks GPU worker activity and stops the GPU EC2 instance if no training or inference has been active for a configured amount of time. To automate this, I configured EventBridge to invoke the idle-check process every five minutes. EventBridge calls Lambda on a schedule, Lambda checks worker activity, and if the instance has been idle long enough, Lambda stops it to save cost.

I also improved model loading and reference audio selection. Selecting a voice profile can now automatically load the matching GPT and SoVITS models, removing the need for users to manually press a “load models” button. I helped fix the first-load issue where users previously had to switch profiles and switch back before the correct model loaded. For reference audio, I helped add automatic selection of a primary reference clip and up to five auxiliary reference clips based on transcript quality, language, file type, and filename signals. Users can still manually override the selection, but the automatic selection makes inference easier and improves the chance of using better reference audio. I also helped add retry logic when fetching generated audio from S3, because sometimes the audio URL appears before the file is fully available.

Throughout Week 5, I also added and updated tests and documentation. This included tests for live conversation helpers, reference audio selection, Lambda instance status/start/idle-check logic, and live gateway WebSocket control behaviour. I verified frontend production builds and targeted backend, Lambda, and live-gateway tests. I also continuously updated deployment documentation because the architecture changed repeatedly based on supervisor requirements. Overall, Week 5 was mainly about transforming the project into a more complete, cloud-ready, secure, and cost-aware system with Lambda for REST APIs, GPU EC2 for training/inference, live gateway for chatbot WebSocket communication, S3 for shared storage, CloudFront and ALB for routing, and EventBridge for automated cost control.

Week 6 – Sprint 6 Contributions

During Week 6, my main contribution was maintaining and improving the deployed cloud system while adding multilingual support, improving security, documenting the full architecture, solving GPU storage issues, and preparing the project for a split deployment structure. While my groupmate worked on Speech Synthesis Markup Language support, I focused on keeping the overall system working, secure, updated, documented, and aligned with supervisor feedback. A lot of this work involved repeated rebuilding, redeployment, testing, and replacing earlier approaches as the deployment plan changed, so part of my contribution was continuously adapting the system without losing the existing training and inference functionality.

One major Week 6 task was fixing the secured CloudFront-to-Lambda Function URL flow. Because the Lambda Function URL was protected using AWS IAM authorization, requests needed to be properly signed using AWS Signature Version 4. I helped identify that POST and other non-GET requests were failing because the required `x-amz-content-sha256` header was missing. This header is needed for request signing and payload verification. I updated the request flow so JSON requests include the required payload hash header, and I also updated Lambda CORS settings to allow `x-amz-content-sha256`. This allowed the browser to pass CORS preflight checks before sending the actual request. I then continued debugging CloudFront origins, behaviours, OAC settings, CORS rules, and Lambda permissions until the deployed website worked correctly with the secured backend. The final setup allows users to access the backend through CloudFront while the raw Lambda Function URL only accepts properly signed CloudFront-origin requests.

During Week 6, I also helped present the website during scrum. The website was marked as completed, with some minor remaining changes such as multilingual support. I also researched and documented alternative deployment approaches, especially whether SageMaker could replace or improve the EC2-based GPU setup. I looked into SageMaker real-time inference, serverless inference, training jobs, autoscaling, and whether training and inference should be split into different job types. Although SageMaker provides managed ML features, I documented that it may be more costly and less suitable for our current live voice use case compared to GPU EC2 with controlled start/stop behaviour and idle shutdown. This helped justify the current approach of using GPU EC2 for heavy workloads while reducing cost through Lambda start control and EventBridge idle shutdown.

Another major Week 6 contribution was adding Chinese support to the Live voice chat flow. I implemented an explicit English/Chinese language selector so users can choose the conversation language instead of relying only on automatic detection. The selected language is passed from the frontend through the WebSocket into the OpenAI Realtime session, allowing transcription and assistant replies to stay in the selected language. I also updated the GPT-SoVITS TTS request logic so Chinese mode sends `text_lang: all_zh`, which better matches GPT-SoVITS language handling and avoids unnecessary mixed-language detection. I improved Chinese punctuation splitting so Live Fast can split Chinese assistant replies into phrases more naturally.

While debugging Chinese Live Fast issues, I traced failures across the frontend, Lambda, GPU worker, and GPT-SoVITS API. I improved GPU worker error handling so it surfaces the real GPT-SoVITS exception details instead of only returning a generic “tts failed” message. This made debugging much easier because the actual failure reason became visible. I also found that Chinese replies containing English words or English-style number normalization could trigger language detection issues. To reduce this, I helped skip English text preprocessing in Chinese mode and sanitized Chinese TTS payloads to remove Latin-script text before sending the text into GPT-SoVITS. I added and updated tests for language mapping, Chinese punctuation splitting, Chinese TTS sanitization, OpenAI Realtime language configuration, and language-aware preprocessing, then verified the frontend build and relevant service tests.

A major task in Week 6 was creating high-level system documentation for the supervisor. The supervisor wanted a clearer explanation of how the whole system works, not only code-level details. I helped document how the frontend calls cloud services through CloudFront, how each endpoint is routed, what each service is responsible for, how Lambda connects to the GPU worker, how WebSocket and SSE traffic are handled, where S3 is used, how trained models and generated audio move through the system, and what security rules exist. This documentation also explained the responsibilities of CloudFront, Lambda Function URL, ALB, GPU worker, live gateway, S3, EventBridge, EC2, and security groups. Since the deployment structure changed many times due to supervisor requirements, this documentation became important for helping the team understand and maintain the current architecture.

I also helped make testing more flexible by allowing a local Vite frontend server to connect to the deployed cloud services through CloudFront. Instead of only supporting fully local testing or fully deployed testing separately, the frontend can now run locally in the browser while still using deployed Lambda, GPU worker, and live gateway services. This required CORS and environment configuration changes so the local frontend origin could communicate correctly with the cloud backend. This made development faster because frontend changes can be tested locally without redeploying the whole site every time. I also fixed minor bugs and user-experience issues based on supervisor feedback, including reducing confusing frontend behaviour and improving error/noise handling.

Another important Week 6 issue was GPU EC2 disk usage. I investigated a supervisor-reported GPT training failure and traced it to local disk exhaustion on the EC2 instance rather than missing S3 storage. I confirmed that final trained models are stored in S3, but GPT-SoVITS still writes temporary training data, extracted features, logs, and PyTorch Lightning checkpoints to local EC2 disk during training. This means local disk can still fill up even when S3 is used correctly. I helped clean the GPU EC2 instance, reduced disk usage significantly, identified which directories were safe to remove, and separated required runtime/pretrained files from disposable experiment artifacts.

After solving the immediate disk problem, I helped implement GPU worker cleanup improvements. After a successful training run, the worker uploads the final GPT and SoVITS model artifacts to S3 first, then removes local temporary experiment folders and logs. I added safety checks so cleanup only targets the expected training scratch and log areas, preventing accidental deletion outside the correct workspace. Cleanup does not run during active training because temporary training state is still needed while training is running. On the operations side, I also helped set up and document a daily EC2 cache refresh policy that clears local model caches and local weight folders, so the next model selection reloads from S3 instead of keeping stale or bulky local files. I documented test commands and verification steps so future debugging can prove that local disk is being cleaned while S3 remains the main storage location.

During Week 6, I also completed the Microsoft AI Agent-a-thon on behalf of the company. This was an online guided programme focused on Microsoft’s AI ecosystem and AI agent development. The programme included live teaching, Q&A, guided lessons, tutorial videos, practical exercises, and a learning platform with a personal AI mentor. I learned how Microsoft 365 Copilot can be used to build a personalised AI agent by defining the agent’s purpose, shaping its behaviour through prompts, providing resources and knowledge sources, and configuring it to respond based on business context. This gave me practical exposure to prompt engineering, agent configuration, knowledge grounding, AI-assisted workflows, and how companies can use Copilot-based agents to improve productivity, employee support, and task automation.

A final major Week 6 contribution was working on the split deployment structure. I created and worked on the deployment-split-change branch to convert the project from one combined frontend into two separate frontend apps: one **Training** web app and one **Live Fast** chatbot web app. The frontend now supports build modes using `VITE_APP_MODE=training` and `VITE_APP_MODE=live-fast`, so each CloudFront distribution can serve only the app it is meant to expose. Training builds into a separate training output, Live Fast builds into a separate live-fast output, and both apps still share the same Lambda, GPU worker, live gateway, and S3 backend.

On the Training side, I kept the existing training quality path intact. I did not redesign the upload, S3 handoff, GPU worker pipeline, training orchestration, retry behaviour, or model upload flow. I added frontend validation so bad inputs are caught earlier, removed frontend-visible training logs, and kept detailed logs as backend output suitable for CloudWatch. Once training is completed and models are uploaded, the Live Fast app can load the trained voice model from the shared backend and S3-backed model list. On the Live Fast side, I focused the app around the chatbot flow. The user selects a trained voice profile, saves it, and the app loads the matching GPT and SoVITS models. Live Fast then loads processed training audio, auto-selects the best primary reference and auxiliary reference clips, and allows the user to override these settings if needed. I also kept important inference controls available, such as speed, top_k, top_p, temperature, and repetition penalty.

I removed the old user-facing Inference tab and Live Full route from the active app shell. The backend inference endpoints still remain because Live Fast depends on them for readiness, model loading, and phrase TTS, but the old full inference page is no longer part of the deployed frontend experience. I also removed the old fake/local Lambda testing path from the normal workflow. Testing is now intended to use the local frontend connected to deployed CloudFront, Lambda, GPU worker, and live gateway services, reducing the risk of testing something that only passes against a fake local backend. I also improved backend logging for training by making GPU worker training logs emit as structured backend logs that can be collected by CloudWatch, keeping noisy logs out of the frontend while still preserving important debugging information.

The split deployment also required solving a CORS issue because the Training app and Live Fast app are served from different CloudFront domains. I helped update Lambda, GPU worker, and live gateway to support comma-separated allowed origins, so both frontend apps can talk to the same backend stack without CORS blocking one of them. I also continuously updated deployment documentation with the split CloudFront routing map, separate S3 origin paths, same-origin frontend environment guidance, CORS configuration for two domains, and clear behaviour ordering for Training versus Live Fast. The final deployment shape is that Training CloudFront serves the Training app and routes training progress plus API requests correctly, while Live Fast CloudFront serves the chatbot app and routes WebSocket traffic to the GPU ALB before normal API traffic goes to Lambda. Both apps share the same S3 bucket and backend resources for training data, trained models, processed reference audio, and generated outputs.

Overall, Week 6 focused on stabilising the production system after major architectural changes. I improved secure Lambda Function URL access, fixed SigV4 and CORS issues, added Chinese language support, documented the full cloud architecture, investigated SageMaker feasibility, cleaned GPU EC2 disk issues, improved post-training cleanup, completed Microsoft AI Agent-a-thon learning, and helped split the frontend into separate Training and Live Fast deployments. This week was less about one single feature and more about making the whole system more secure, maintainable, cost-aware, multilingual, production-ready, and easier for the supervisor to understand.
