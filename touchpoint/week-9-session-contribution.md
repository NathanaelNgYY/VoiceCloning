# Week 9 - Session Contribution

During Week 9, I focused on stabilizing the cloned-voice integration between the VoiceCloning website and the ClinicalChatbot so the other system can depend on a simpler and more reliable backend contract. The main backend improvement was allowing the VoiceCloning API to accept `voiceProfileId + text` directly on the synthesis routes instead of requiring the external system to fetch the full saved profile first and manually pass all cloning parameters each time. I updated the Lambda synthesis flow so the backend can resolve the full saved profile internally, load the correct GPT and SoVITS model pair when needed, apply the saved reference audio, prompt text, prompt language, auxiliary references, and tuning defaults, and then return the cloned audio response. This made the VoiceCloning side behave more like a proper voice service and reduced the amount of integration logic that has to be maintained in the ClinicalChatbot backend.

I also strengthened the backend structure by separating the model-loading and saved-profile resolution logic into shared helper modules instead of keeping those responsibilities spread across multiple handlers. This improved the maintainability of the Lambda code and made the behavior easier to test across both the short Live Fast sentence route and the longer inference route. I added and updated automated tests around the new profile-resolution flow, route behavior, and model-loading support so the new contract could be verified before deployment rather than relying only on manual testing.

Another major part of this Week 9 work was improving how the website keeps its saved voice profile aligned with the selected voice and reference clips. The goal was that when a voice is selected on the website, the saved profile used by external systems should reflect the same primary reference, auxiliary reference clips, and inference defaults rather than staying on an older saved profile. I implemented an automatic profile sync flow so the website can save the currently selected voice setup back to the server after the best reference clips are chosen. This was important because the ClinicalChatbot integration depends on the saved profile data being complete and accurate, especially for reference audio and auxiliary clips, rather than only showing the correct setup temporarily in browser state.

This part of the work required a large amount of debugging and repeated rebuilding because several frontend timing problems appeared once the auto-save behavior was introduced. I investigated cases where the page stayed stuck on a saving state, where the newly deployed frontend crashed in the browser because of variable initialization order, and where switching voices could accidentally reuse stale clip lists from the previously selected voice before the new voice's training clips had finished loading. I then fixed the save-loop behavior, corrected the frontend initialization bug, cleared stale reference state when switching voices, and added source-aware guards so automatic reference selection only happens after the correct training audio list for the selected voice has finished loading. This made the automatic profile update more dependable and prevented incorrect reference clips from being saved.

I also added browser-side debugging support to make profile verification easier during deployment and testing. The frontend now logs the auto-selected reference setup, the activation request payload, the activation response, and any activation error in a structured way so it is easier to confirm whether the primary reference and five auxiliary clips are actually being saved. This was useful because some earlier issues looked like display problems at first, but the logs helped reveal whether the real problem was stale frontend state, premature saving, or the wrong clip list being used during voice switching.

Beyond coding and debugging, I updated the integration and handoff documentation so the current behavior is clear for later continuation. I refreshed the chatbot integration notes to explain that the VoiceCloning backend now supports direct `voiceProfileId + text` synthesis, that the website-driven active profile can be synchronized after reference auto-selection, and that the short sentence route itself does not perform its own sentence-looping logic. I also documented the practical deployment implications of the recent frontend fixes, including the need to redeploy both frontend builds after changing the Live Fast and TTS pages. This helped keep the technical work, testing flow, and deployment expectations aligned across both systems.

Overall, Week 9 was mainly about turning an initially fragile cross-system voice-selection flow into a more dependable service contract. Instead of only making the website appear to use the right cloned voice locally, I worked on making sure the saved profile state, Lambda behavior, frontend auto-selection flow, debugging visibility, and documentation were all consistent enough for the ClinicalChatbot to use the intended cloned voice more reliably in practice.

In a later Week 9 follow-up session, I continued the work from the ClinicalChatbot side so its backend would match the newer VoiceCloning service contract more cleanly. I simplified the voice-routing Lambda so it now resolves the locally bound `voiceProfileId` and forwards only `voiceProfileId + text` to the VoiceCloning sentence synthesis route instead of continuing to act as an adapter that fetches the full profile and manually forwards all cloning fields. This reduced duplicated profile-contract logic between the two systems, removed unnecessary backend coupling, and aligned the chatbot-side integration more closely with the updated server-side ownership model on the VoiceCloning system. I also added regression tests around this direct handoff path and removed no-longer-needed helper logic from the Lambda package so the behavior is easier to maintain and less confusing for future changes.

I also traced the current chatbot frontend playback behavior more carefully so the real runtime flow is now better understood and documented. I confirmed that the temporary voice debug control on the activity page still uses the same backend voice-response route as normal cloned playback, which means it now follows the updated sentence endpoint path as well. I also confirmed that the Gemini live conversation path already detects completed sentence segments and queues them for cloned playback in order, while the OpenAI realtime path still waits for a fully completed assistant reply before queueing cloned voice output. This clarification was useful because it explains the current latency differences between the two conversation paths and shows where future phrase-by-phrase improvements would still be needed. This reduced unnecessary dependence on assistant-audio transcription and aligned the OpenAI path more closely with the speech-to-text to cloned-voice architecture that is better suited to the current chatbot setup.

In another Week 9 session, I focused on an operational reliability issue affecting the cloud deployment: the GPU instance could still be shut down by the automatic idle-stop flow even while the ClinicalChatbot was actively using cloned-voice synthesis. I traced this through the full cost-control path across EventBridge, the Lambda instance-control layer, and the split training and inference worker setup on the GPU side. The main issue was that the shutdown check was only reading activity from the training-side worker, so real cloned-voice traffic reaching the inference-side worker could be missed and treated as idle time. I then updated the status-check logic so the automatic shutdown decision is based on activity from both worker roles instead of only one side of the deployment. This was important because the deployed system had already grown beyond a single-worker assumption, and the cost-control logic needed to reflect the newer split architecture to avoid interrupting active chatbot voice usage.

This session also involved regression testing, packaging, and deployment-oriented verification rather than only changing the code. I added a targeted automated test for the split-deployment case where chatbot synthesis is active on the inference side while the training side appears idle, and I verified that the revised Lambda behavior keeps the GPU running in that scenario while still allowing proper shutdown after genuine inactivity. To make the fix work cleanly with the existing shared routed deployment, I also added the necessary inference-side activity endpoint and rebuilt the Lambda deployment package for release. Overall, this work improved the reliability of the automatic GPU shutdown mechanism, reduced the risk of active cloned-voice sessions being interrupted incorrectly, and strengthened the operational side of the VoiceCloning and ClinicalChatbot integration.

In another Week 9 session, I focused on improving the stability of the ClinicalChatbot startup flow during live OpenAI Realtime conversations while also documenting the current application structure more clearly. I reviewed and refined the system explanation so the learner-facing and instructor-facing views are easier to understand, including the learner dashboard, assignment list, pre-brief page, live activity page, results page, instructor dashboard, and case management screens. The main technical issue I investigated in this session was an unreliable first-message problem during the live consultation flow. I traced this to the startup sequence in OpenAI Auto mode, where microphone activation, voice activity detection, and the forced initial assistant greeting were all happening before the application had confirmed that the learner had actually spoken. This made the first transcript more vulnerable to startup noise, room sound, or unclear audio, which could then be treated as an unintended doctor turn and cause the first patient reply to sound unnatural or mismatched.

To improve this, I implemented a startup audio safeguard on the ClinicalChatbot side. The browser microphone now uses more stable audio constraints, starts in a muted state during the initial OpenAI greeting, and is only enabled after the greeting playback has finished and the session is ready to listen for the learner properly. I also added a filtering step so startup noise-only transcripts and placeholder outputs such as unintelligible text are ignored until the first real learner utterance is confirmed. To support this change, I added focused automated tests around the startup guard behavior, re-ran the related realtime session and grading tests, and verified the frontend production build so the fix could be checked more systematically before further live testing.

This makes the first chatbot reply faster after GPU startup because the worker no longer has to wait until the first synthesis request to fetch the reference audio and auxiliary clips.
