# Week 11 - Session Contribution

This week, I improved the reference audio and configuration management flow for the voice cloning system. The main focus was making trained voice models easier to compare, reproduce, and tune after training. Previously, the system could auto-select reference clips, but the selection was mostly hidden from the user and did not clearly show why one clip was selected over another. I upgraded this into a stricter reference selection flow that filters for better clips first, then ranks them with visible scoring. The UI now shows reference scores, strict/manual status, and rejection reasons such as clips being too short, too long, missing sentence-ending punctuation, or not meeting the preferred reference criteria.

I added metadata tracking for both training and inference. Training runs now save metadata such as the engine version, denoise setting, epochs, batch size, selected references, and source dataset statistics. This is important because good voice cloning results depend heavily on the training settings and reference clips used, so saving these details makes it easier to reproduce or compare successful runs later. On the inference page, the system now shows training metadata separately from inference settings, so the user can clearly see what was fixed during training and what can still be tuned during inference.

I also built a per-person configuration system for trained voice models. Each voice profile can now have multiple saved configurations. A configuration stores the selected primary reference clip, auxiliary clips, inference settings such as speed, top-k, top-p, temperature, and repetition penalty, plus training metadata for context. This allows different combinations of references and inference settings to be saved and compared without losing previous work. The configuration list is saved in S3 under each voice profile, so the settings are kept after page refresh and can be reused later.

The configuration UI was improved through several iterations. I added the ability to save new configs, update only the currently loaded config, delete configs, rename configs inline, and reorder configs by drag-and-drop. The first ranked config is treated as the active preferred setup. When the top config changes or is updated, it syncs back into the voice profile so that the voice profile remains the source of truth for the external chatbot and the main voice cloning system. This is important because the external chatbot refers to voice profiles, including assignment-related profile identities, so configs should remain variants under the voice profile rather than replacing it as the runtime source of truth.

I added short sample generation for saved configurations so users can listen to how a specific config sounds. The sample uses that config's reference clips and inference settings. This makes it easier to compare different metadata and reference combinations by hearing the output directly. I also fixed sample playback issues caused by temporary browser audio URLs being cleared too early, and added clearer error handling when a config is missing the reference transcript needed for sample generation.

I spent a significant amount of time debugging the new configuration flow after it was connected to the deployed environment. One issue was that the new config API worked locally but returned server errors in the browser because the Lambda deployment package did not include the new config module, so I fixed the packaging script and rebuilt the Lambda zip. Another issue was CloudFront rejecting some request methods, so I changed config save and delete actions to use POST-compatible requests. I also debugged the generated sample audio player, where browser blob URLs were being revoked too early and caused playback errors. Later, I fixed UI state bugs where the loaded label could stay on the wrong config after reordering, and corrected the architecture so the voice profile remains the source of truth while configs act as saved variants under it.

On the backend, I added API support for saving, listing, updating, deleting, and loading per-person configs. I also added a training metadata endpoint so the inference page can fetch saved run metadata from storage. The Lambda packaging script was updated to include the new config module, and the config save/delete calls were adjusted for CloudFront compatibility. I also added console logging for config load, save, delete, reorder, sample generation, and training metadata loading, so it is easier to debug whether the config system is working correctly in the browser.

I clarified and corrected the source-of-truth design during the session. The final design keeps the voice profile as the authoritative record used by synthesis and the external clinical chatbot. Configs are saved experiments under that voice profile. When a config becomes the top-ranked setup, it updates the voice profile, but backend synthesis still reads from the voice profile rather than directly overriding it from config storage. If a voice profile has missing or weak references, the backend can auto-select references and create a default config record for visibility.

Testing and verification were also part of the work. I added and updated targeted tests for reference selection, voice profile metadata, training metadata, config APIs, routing, and backend voice profile resolution. I repeatedly ran frontend production builds to catch UI and JSX issues, and rebuilt the Lambda deployment package after backend changes. The remaining work is to redeploy the updated frontend and Lambda package, then browser-test the full flow on the deployed system with real trained models.

I also continued debugging the deployed Live Fast configuration flow after the first round of fixes. The default configuration was being saved, but the active voice profile still failed to update in the browser. I traced the problem through the frontend, Lambda routes, saved config API, and deployed CloudFront path instead of assuming it was the same bug. This showed that config saving was already working, but the voice profile activation request was being blocked before it reached Lambda when the request body became too large. The large body came from sending full training, reference scoring, transcript, and inference metadata together during profile activation.

To fix this, I separated the purpose of saved configs from the purpose of the active voice profile. Saved configs continue to keep the richer metadata needed for inspection, comparison, and reproducibility. The active voice profile now sends a smaller metadata summary during activation, keeping the selected reference paths, basic scores, training context, and inference defaults while removing bulky reference breakdowns and transcript details from the activation request. This keeps the voice profile compatible with the external chatbot and synthesis flow while avoiding CloudFront request blocking.

I verified the fix with targeted tests and a deployed request check. I added a test to ensure voice profile activation payloads are compact enough for the deployed path, rebuilt the Live Fast frontend successfully, and confirmed that a compact real activation payload passed through CloudFront and returned a successful response. This completed the main reliability issue: the best ranked config can be synced into the saved voice profile, and chatbot or external chatbot inference can continue using the saved voice profile as the runtime source of truth.
