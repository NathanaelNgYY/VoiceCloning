# Week 12 - Session Contribution

This week, I worked on a focused research and evaluation task: deciding whether the A*STAR MERaLiON API gives better speech recognition than the self-hostable open-source MERaLiON model, specifically for recognising drug names in a medical-education setting. The decision driving the work was practical. If the open-source model matches the hosted API, there is no quality reason to depend on A*STAR, but if the API is meaningfully better, then engaging them would be worthwhile. My goal for the week was to turn that question into a fair, evidence-based comparison rather than an opinion.

I began by investigating what the MERaLiON API actually serves. Using the API's own model-listing and documentation endpoints, I confirmed the base service, verified that our access key sits on an unlimited usage tier, and recorded the exact list of models the API exposes. This already answered part of the question, because the API serves the same open model checkpoints that are published publicly, alongside a few additional speech recognition models that are not downloadable. I then cross-checked each model against the public Hugging Face repository to establish which ones can be self-hosted and which are API-only, since the API-only models were the only realistic way the hosted service could outperform self-hosting. I also confirmed the exact request and response format for the transcription endpoints so that every test would call the models correctly and consistently.

I built a small evaluation harness so the comparison could be measured rather than judged by impression. I created a drug-name test set of clinical sentences that deliberately includes confusable drug pairs, such as metoprolol against metformin and hydralazine against hydroxyzine, because the dangerous real-world failure is one real drug being mis-heard as a different real drug. Each sentence was paired with a known correct transcript, the drug name being tested, and a pronunciation guide so the audio could be read consistently. The harness was designed to support two audio sources that feed the same scoring system, so the results would be directly comparable.

The first source is a synthetic one that generates spoken audio across five English accents, including American, British, Indian, Singaporean, and Australian, using text-to-speech. This let me scale the test quickly and check accent sensitivity. The second source is a human one. I built a simple local web application where a person can read each sentence aloud, record it directly in the browser, and have it transcribed live. All recorded audio is automatically converted to the 16 kHz mono format the API requires, so the user does not need to worry about recording settings, and the same recordings are saved so they can be scored later.

For scoring, the harness measures word error rate and character error rate, drug-name recall, and a dangerous-swap log that flags cases where a missed drug was replaced by a different valid drug name. It also runs a paired statistical significance test so differences between models are reported as significant or not, rather than as raw percentages that could be misread. I separated the scoring step from the transcription step, so reports can be regenerated instantly from saved transcripts without repeatedly calling the API. I spent time debugging the harness as it came together, including handling the model outputs that contain speaker tags, fixing a console display issue on Windows, and recovering a run that had been interrupted partway by scoring directly from the saved transcripts.

I then ran the full synthetic evaluation, covering 190 clips and 210 drug mentions across all five accents and all five models. The results were clear and consistent. The best model overall was MERaLiON-2-10B, which is self-hostable, with about 85 percent drug-name recall. Importantly, the open self-hostable models and the API-only experimental speech model produced identical results on every clip, which strongly indicates that the API routes these model names to the same underlying model. The remaining API-only models performed worse, not better. Every model still produced some dangerous drug-to-drug swaps, and recognition was weaker on certain accents, which confirmed that accent sensitivity is real. The practical conclusion is that there is no speech-recognition quality reason to depend on the A*STAR API, because self-hosting MERaLiON-2-10B achieves the same quality, and the real improvement lever is domain adaptation and a human verification step rather than the choice of API against open source.

Alongside the technical work, I authored a written comparison document, MERaLiON_ASR_Comparison.docx, which captures the comparison approach, the findings, and the recommendation in a form suitable for sharing with the team. The evaluation harness also writes a structured findings report automatically for each run, so the results stay reproducible and can be re-generated as more data is collected.

A few items remain open and are noted for follow-up. These include confirming the MERaLiON licence terms for commercial and clinical-education use, clarifying ownership of work produced during the internship, and recognising that human verification of any drug name is required for any patient-facing use regardless of which model is chosen. The immediate next step is to run the human voice track through the same harness and compare it against the synthetic results, since synthetic speech tends to pronounce drug names more cleanly than real human speech and may give slightly optimistic numbers. Overall, this week produced a complete and reproducible evaluation that answers the core question with evidence, supported by the test harness, the synthetic results, and the written comparison document, with the human-voice comparison set up and ready to run next.

In a separate strand of work this session, I extended the same evaluation thinking to a second candidate model, Google's MedASR, so that MERaLiON would not be judged in isolation. The underlying driver is the supervisor's actual requirement: a transcription model that can reliably and accurately recognise a vocabulary of roughly three thousand drug names. Because that requirement is specifically about medical terminology rather than general speech, it made sense to bring in MedASR, which Google has fine-tuned for healthcare dictation, and to measure how reliable it really is on this task before assuming any one model is the answer. The point of the exercise is not to crown a winner prematurely but to gather comparable statistics and behavioural information so that MERaLiON and MedASR can be evaluated against each other on the same footing, and so the supervisor's question can be answered with numbers rather than impressions.

To do this I built a small MedASR accuracy lab mirroring the structure of the MERaLiON harness, so the two efforts produce comparable evidence. It has two complementary tracks that feed one shared scoring system. The first is a local web application with two modes: a normal mode that presents shuffled everyday sentences, and a medical mode that presents drug names, optionally with dosage and frequency phrasing. The user reads each prompt aloud, records in the browser, and the clip is converted to sixteen kilohertz mono audio, transcribed by MedASR, and scored, with the test auto-advancing through the set and a full analysis produced at the end. The second is a synthetic track with no interface, which generates spoken audio across multiple English accents using text-to-speech, transcribes it with MedASR, and reports the same metrics. For this comparison the synthetic track is configured to run a fixed number of clips per accent across British, Indian, Singaporean, and American voices, and the analysis breaks the scores out per region so accent sensitivity is visible rather than averaged away.

Scoring reuses the word-error-rate approach from MedASR's own reference notebook, reporting corpus and per-item error rates together with an exact-match rate that is especially meaningful for single drug names. One important correction came out of early results: MedASR is a dictation model and spells spoken punctuation out as words such as period and comma, which our prompts never contain, so those words were inflating the error rate and making a perfectly transcribed drug name look completely wrong. I added a normalisation step that strips this dictation punctuation before scoring, which restored sensible numbers, and confirmed the change on worked examples. Early observations also matched expectations for a domain-specialised model: MedASR degrades on general everyday sentences, sometimes hallucinating medically flavoured words, while the medical drug-name mode is where its specialisation should pay off. That contrast is exactly the signal the comparison is meant to capture. The next step on this strand is to run the full hundred-clip normal and medical evaluations across the four chosen accents and place MedASR's drug-name reliability side by side with the MERaLiON results, keeping the supervisor's three-thousand-drug-name requirement as the benchmark both models are ultimately measured against.

Another contribution this week was improving the voice cloning website so it can support a client-facing text-to-speech use case, not only chatbot interaction. The practical motivation came from a full-time team member who had a client interested in trying his cloned voice. The client provided both a script and an audio sample, and we were able to clone the voice, but the existing website experience was mainly built around live chatbot interaction. That meant the user could talk to the chatbot and hear cloned responses, but there was no simple way to paste a prepared script and generate a finished speech recording from the cloned voice. To address this, I added a Text to Speech tab to the Live Fast website so a script can be passed directly into inference and returned as downloadable speech audio.

The new Text to Speech page reuses the same voice profile, model loading, and saved inference configuration already used by the Live Fast chatbot, so the generated script audio uses the same selected cloned voice and tuning settings instead of creating a separate or inconsistent flow. I added two generation paths: a Live Fast path for short sentence-style generation and a Full Inference path for the longer inference route. This lets us compare fast output against fuller synthesis while still using the same voice configuration. I also made the page cloud-ready by exposing the tab through a CloudFront-friendly query route, so it can be tested from the deployed Live Fast site without depending on special single-page-app fallback routing.

I then improved the generated-audio handling after testing the workflow. Instead of replacing the previous output every time a new script is generated, the browser now keeps a local history of generated clips for both Live Fast and Full Inference outputs. Newer clips appear at the top, and each generated result remains playable and downloadable during the browser session. This is useful for client testing because multiple script attempts can be compared without losing earlier versions. The generated files are intentionally stored only in the browser for now and are not uploaded to S3, which keeps the feature lightweight while still allowing the team to produce and download speech recordings for review.
