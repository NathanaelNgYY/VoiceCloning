# Week 8 - Session Contribution

During Week 8, I focused on integrating the deployed voice cloning system with the ClinicalChatbot so that each chatbot can gradually be matched to a selected cloned voice instead of relying on the default built-in voice. This work required me to study both systems more carefully, understand how the existing frontend, backend, Lambda, API Gateway, and CloudFront pieces were already connected, and compare multiple architecture options before settling on a more maintainable approach. At the beginning of this stage, the systems were not yet ready for each chatbot or character to have its own long-term cloned voice identity. The earlier flow depended more on a globally active voice concept, which was not suitable if different chatbot personas needed different cloned voices. Through this iteration, I refined the design toward a per-chatbot `voiceProfileId` approach, which is a better mid-stage solution because it allows me to manually map the correct cloned voice to each person while keeping the full voice configuration on the backend side.

On the ClinicalChatbot side, I updated the frontend logic so it can work with backend-owned voice routes rather than relying only on the original built-in speech path. I added the supporting request flow for retrieving the selected voice profile and requesting a generated cloned-voice reply after the chatbot produces its text response. I also improved the frontend runtime configuration so the system can communicate with configurable backend endpoints during deployment and testing. This part of the work was not only about coding new requests, but also about understanding the existing codebase in more depth, tracing how the current assignment APIs, authentication flow, and realtime conversation flow were already working, and then fitting the new cloned voice integration into that structure without disrupting the rest of the chatbot behavior.

On the backend and cloud side, I prepared a dedicated Lambda-based voice orchestration layer for the ClinicalChatbot and worked through the practical deployment setup needed to expose it safely. This included packaging and preparing the new Lambda, reviewing how a Function URL could be used as a direct backend entry option, and then refining the final approach so the chatbot can continue using its existing API structure while API Gateway routes the new voice endpoints to the dedicated Lambda. I added the necessary API Gateway resources and methods for the new voice-related routes, aligned them with the existing authorization pattern, and updated the CORS configuration so the frontend can call the new endpoints correctly from the browser. I also spent time learning the operational side of API Gateway setup, including how resources, methods, authorization, `OPTIONS` handling, deployment stages, and browser preflight behavior interact in practice when adding a new Lambda-backed feature.

I also coordinated this work with the other voice cloning system so the ClinicalChatbot backend can communicate with it more cleanly. Instead of exposing full cloning configuration to the browser, the design now keeps the detailed profile data on the backend side and uses protected backend-to-backend calls to retrieve the correct full profile by `voiceProfileId`. I worked through the earlier design limitations, compared direct-calling and orchestrator-style approaches, and refined the structure into a dedicated ClinicalChatbot Lambda that can resolve a local voice binding, request the correct saved voice profile from the other system, and then trigger the appropriate synthesis route there. This helped move the project from an earlier stage where a selected voice did not reliably belong to a specific chatbot, toward a more organized mid-stage setup where I can manually assign a cloned voice profile to each chatbot or person and have the system use the intended voice more consistently.

This Week 8 work involved a lot of architecture review, backend integration planning, deployment debugging, and repeated refinement rather than only writing isolated frontend code. I had to compare different routing strategies, understand where CloudFront routing was useful and where API Gateway was the better fit, review how Lambda Function URLs differ from API-managed routes, and trace how the frontend should communicate with the backend without breaking the rest of the ClinicalChatbot APIs. Although the current stage still uses manual voice-profile binding rather than a fully automated profile-management flow for every chatbot, the system is now in a much stronger position for future expansion because the cloned voice integration is being shaped around clearer backend ownership, safer API design, and a path toward giving each chatbot persona its own consistent cloned voice identity.

I then continued this Week 8 work on the VoiceCloning system itself so it could support the ClinicalChatbot more reliably from the backend side. I implemented server-side saving of cloned voice profiles so the selected GPT-SoVITS model pair, reference audio, prompt settings, auxiliary references, and tuning defaults could be stored in shared cloud storage instead of remaining only inside frontend state. This made it possible for the other system to depend on a stable `voiceProfileId` contract instead of trying to reuse temporary browser-selected data. I also refined the integration model so the ClinicalChatbot can keep ownership of which chatbot or assignment should use a given `voiceProfileId`, while the VoiceCloning side keeps ownership of the full synthesis profile and returns only the correct backend data when requested.

As the requirements became clearer, I compared several possible approaches before settling on the more maintainable direction. An earlier idea was to let the other system follow whichever voice was globally active on the website, but this would not remain correct if different chatbot personas needed different cloned voices over time. Through iteration, I improved the design toward a split responsibility model where the chatbot side manages the assignment-to-voice binding and the VoiceCloning side resolves the complete saved profile by `voiceProfileId`. To support this, I added an internal backend-facing profile lookup route, kept the public route limited to summary data only, and documented the full integration flow so another development session could continue the work with a consistent source of truth.

This session also involved a significant amount of debugging and deployment validation rather than only feature coding. I traced build problems in the frontend, fixed import and integration issues that affected deployment builds, and verified the separate training and live-fast frontend outputs after the new compatibility changes were introduced. I also investigated repeated `502 Bad Gateway` behavior on the new voice-profile save route even after the Lambda logic had been updated. That debugging showed that the deployment package itself was missing the new voice-profile handler folder, so I corrected the Lambda packaging script and rebuilt the deployment artifact. After that, I verified the route packaging more carefully and improved the frontend so the currently active `voiceProfileId` is shown directly in the interface instead of only appearing in network responses or temporary save messages. This made the feature easier to validate during testing and helped confirm that the website and backend were finally saving and exposing the selected profile correctly.

Overall, this part of Week 8 strengthened both the architecture and the day-to-day usability of the integration work. Beyond adding backend routes and frontend controls, I had to repeatedly adapt the implementation as new constraints became visible, such as the difference between a global active voice and a stable per-chatbot voice identity, the need to protect internal profile lookups, the impact of CloudFront and Lambda deployment packaging on route availability, and the importance of showing the saved profile clearly during testing. This iteration-heavy work helped move the project closer to a more dependable cross-system cloned-voice workflow rather than a one-off proof of concept.

I also spent part of this Week 8 work debugging the end-to-end voice response path using direct Lambda tests and API Gateway tests with events and request body (like postman) before relying on the deployed frontend. This helped confirm that the backend routes could resolve the correct saved voice profile and return an audio response in isolation, which made it easier to separate frontend issues from backend and infrastructure issues. To support faster validation on the chatbot side, I added a temporary mock-style test control inside the chatbot activity page so I could type sample text and trigger the cloned voice response directly without depending on the full live conversation flow. That small debugging tool exposed several practical integration problems earlier, including deployment mismatches, API routing issues, frontend response handling, and browser playback behavior, and it helped fix many of the blocking issues before moving back to full chatbot testing. This was done because open ai usage was at its limit and we had to find alternative ways to test before usage is replenish, saving time for future errors and fixes

In a later Week 8 session, I continued the chatbot integration work by exploring how to keep the live conversation usable after the original OpenAI realtime usage was exhausted. I studied whether the ClinicalChatbot could switch to Gemini for the live conversation layer while still keeping the cloned-voice output from the existing VoiceCloning system. This required tracing the selected model flow, checking which frontend paths still defaulted to OpenAI, and confirming that the Gemini browser flow depended on a separate backend token endpoint rather than only a frontend API key change.

I then focused on making sure the chatbot still responds with the cloned voice instead of Gemini's native voice. To do this, I separated the live conversation layer from the speech output layer. Gemini Live is now used for live conversation, transcript handling, and assistant response generation, while the actual audible reply is still sent through the ClinicalChatbot voice-response route. That route goes through the Lambda voice orchestration layer, resolves the correct `voiceProfileId`, and calls the VoiceCloning backend to generate the cloned audio. This was important because it preserved the intended Obama cloned voice behavior even after the conversation model changed.

I also updated the voice-routing Lambda so the chatbot uses the faster sentence-based synthesis route more consistently, and I added more debugging and verification around this path. I used a temporary voice test control in the chatbot page to confirm that the cloned voice route was working independently from the full live conversation flow, which helped separate frontend playback issues from backend synthesis issues. This debugging included CORS handling on the Gemini token flow, a temporary 500 issue from the token endpoint, runtime errors during Gemini session startup, repeated WebSocket close behavior, silent playback cases, and checks to ensure the assistant transcript was correctly forwarded into cloned voice output. This made it easier to prove that the chatbot text, voice routing, and cloned audio output were all connected correctly.

Another important part of this session was improving response speed for longer chatbot replies. At first, the frontend still waited for the full Gemini response before sending it for cloning, so the sentence-based backend route could not provide its intended low-latency benefit. I reworked the live transcript handling so completed sentences can be detected earlier and passed into the cloned-voice queue in order. This means one sentence can start synthesis and playback while later sentences are still being processed. I also strengthened the frontend playback flow and added targeted regression tests for transcript finalization, queued audio behavior, and Gemini live integration. Overall, this session improved both the reliability of the cloned-voice chatbot response path and the practical real-time behavior of the system.
