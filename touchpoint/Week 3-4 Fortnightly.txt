Week 3 – Sprint 3 Contributions

During Week 3, a major part of my contribution was split across two parallel directions of development. One direction was focused on an alternative version of the website built mainly for comparing inference behaviour and improving cloned voice accuracy, while the other direction focused on UI/UX improvements for the main version currently being used. In addition to this, I also made substantial progress on the AWS cloud deployment and distributed system setup, which became one of the most important parts of my work for the project.

For the alternative website version, I worked on improving the long-text inference pipeline so that generated cloned speech would sound less obviously stitched together. Previously, long text could sound fragmented, overly robotic, or unnatural because chunks were broken too aggressively and then joined too mechanically. I improved the chunking behaviour so that the system would no longer flush too early on weak punctuation such as commas or dashes, while still respecting strong sentence endings like full stops, question marks, and exclamation marks. I also added logic to recombine tiny chunks so that the output would not end up as many short fragments. On the audio joining side, I refined how chunk loudness, trimming, pauses, and fades were handled. Instead of flattening everything, the system now only adjusts chunk gain when a chunk is significantly too loud or too soft, trims chunk edges to cleaner waveform boundaries, and applies punctuation-aware pauses and crossfades so that commas receive shorter joins while sentence endings sound fuller and more natural. I also improved the fallback path so that when a chunk fails, the system splits it more intelligently and rejoins the results using the same smoother joining logic instead of hard-cutting audio. This experimental version was mainly focused on accuracy and inference quality, rather than appearance.

Within that same alternative version, I also implemented inference presets such as Studio, Balanced, and Flow. These were not just cosmetic options, but actual inference-time controls that change chunk size, sentence count, join behaviour, and retry strategy. Studio was designed to be the safest and most conservative mode for stable cloning, Balanced provided a middle ground, and Flow allowed longer chunks with lighter joins for smoother flow. This gave more control over how the cloning pipeline behaves depending on whether the priority is stability, naturalness, or speed. I also added support for auto-filling the reference transcript during inference to streamline the workflow and reduce manual friction. Altogether, this version two website was more about refining the speech generation process and giving more structured control over cloning behaviour.

Separately, for the main version currently being used, my Week 3 work was more focused on UI/UX improvements rather than core inference changes. I redesigned parts of the training and inference pages so that the application feels cleaner and easier to use. I changed model selection from raw file-based selection into a more intuitive person-first flow, while still preserving the ability to manually override GPT and SoVITS model choices when needed. I also improved how the app remembers user state by saving inference state, saved drafts, and saved reference sets, which made the workflow much more practical during repeated testing. In addition, I refined audio preview behaviour for reference and auxiliary clips, prevented generated audio from autoplaying unexpectedly, improved status handling and progress restoration, added small popup notices for clearer feedback, and cleaned up navigation, layout, and the training pipeline display. So for Week 3, it is important to distinguish that the long-text and inference-quality work belonged mainly to the alternative experimental version, while the main currently used version focused more on interface and workflow improvements.

Another major part of my Week 3 contribution was the cloud deployment and AWS infrastructure work, which should be emphasized strongly because it was one of the biggest technical efforts during this phase. I worked on moving the project from something that mainly functioned on a local machine into a cloud-ready distributed system on AWS. The deployment architecture I worked with followed the structure of Frontend on S3 and CloudFront, Backend on EC2, GPU worker on a separate GPU EC2, GPT-SoVITS running locally on the GPU machine, and S3 acting as shared storage between services. This was not just a straightforward hosting task. A lot of work was required because the original project assumptions were still too local-development oriented, and the system had to be adjusted so that backend, GPU worker, and GPT-SoVITS could function correctly as separate services with clearly defined responsibilities.

A large portion of this deployment work involved making the system cloud-ready from a configuration and runtime perspective. I helped ensure that the project no longer depended on hardcoded local settings and instead used environment variables for things such as ports, paths, runtime folders, CORS settings, API URLs, and inference ownership. The backend was adjusted so that it could support production deployment more cleanly, including serving the built frontend when needed, supporting Linux-based AWS instances more safely, and exposing health endpoints like /healthz and /readyz so that readiness and liveness checks could be performed reliably. Cleaner shutdown behaviour and safer validation were also part of the cloud-readiness effort. These changes were important because cloud deployment is not just about uploading code; it requires the application to behave correctly under remote runtime conditions, controlled startup, and distributed service ownership.

I also worked on fixing important split-host architecture issues in the deployment. One of the main problems was that parts of the backend still behaved as if GPT-SoVITS was running locally on the same machine, even when the intended architecture used a separate GPU instance. Based on the deployment structure, I helped ensure that in remote inference mode, the backend no longer assumes ownership of GPT-SoVITS and instead behaves as a coordinator that forwards jobs to the GPU worker. The GPU worker became the actual owner of inference operations, including model loading, runtime control, and execution of TTS-related tasks. This was an important architectural correction because it clearly separated responsibilities: the backend handles API orchestration and S3 interaction, the GPU worker owns inference lifecycle, and GPT-SoVITS stays local on the GPU EC2. That separation is a core part of the deployed architecture.

On the AWS side itself, I contributed heavily to provisioning and configuring the required cloud services. This included creating and configuring the Amazon S3 bucket, setting up the CloudFront distribution for frontend delivery, creating the EC2 instances, placing them into the appropriate VPC, subnets, and security groups, and configuring the target group and related network setup. The S3 bucket was not only used for frontend static hosting, but also as centralized shared storage for reference audio, generated audio, model files, and training datasets. This shared-storage design was especially important in the distributed deployment because it allowed the backend and GPU services to exchange files without depending on direct file access on the same machine.

I also worked on the EC2 provisioning and runtime setup itself. The backend side used an Ubuntu-based EC2 instance running the Node.js and Express server, while the GPU side used a dedicated GPU EC2 instance for the gpu-worker and GPT-SoVITS runtime. This required significant setup effort because both machines had different responsibilities, dependencies, and environment configurations. On the CPU/backend EC2, I cloned the project, configured the backend environment, installed dependencies, and ran the backend successfully. On the GPU EC2, I configured and installed the gpu-worker and GPT-SoVITS, including the required runtime settings and startup commands. A lot of the real work here was in handling Linux environment differences, dependency setup, runtime paths, and ensuring that the correct services were running on the correct machine rather than mixing responsibilities.

Network architecture was another major part of the Week 3 cloud contribution. I worked with the VPC design, subnets, Internet Gateway, NAT considerations, security groups, and internal service routing, which are all essential for making a distributed AWS deployment actually function. The deployment document describes a single VPC with a defined CIDR block, public-facing entry through CloudFront and the Application Load Balancer, and isolated backend and GPU roles behind controlled networking boundaries. I also worked through the fact that security groups are directional, which turned out to be an important practical lesson. It was not enough to only think about GPU inbound rules; the backend outbound rules to the GPU worker also had to be correct for communication to work. The final verified internal port ownership was clear: port 3000 for the backend, port 3001 for the gpu-worker, and port 9880 for GPT-SoVITS locally on the GPU instance. This was an important part of stabilizing the deployed flow.

I also contributed to the ALB and target group setup, ensuring that API traffic could be routed safely into the backend and that health checks used the readiness path correctly. The ALB acted as the public backend entry point, while the target group forwarded traffic to the backend EC2 on port 3000. Health and readiness verification formed a meaningful part of the work as well. I validated backend health endpoints, GPU worker health access, GPT-SoVITS local accessibility on the GPU instance, and the corrected backend-to-GPU networking. This testing was important because it confirmed that the architecture was not just documented theoretically, but functionally working end to end.

Lastly for Week 3, I also produced and refined the deployment structure and configuration documentation. This was an important contribution because the deployment work was quite complex, and documenting the architecture, service responsibilities, environment configuration, port mappings, health checks, storage design, security considerations, and deployment limitations made the system easier to understand, justify, and maintain. The document captured not only what was deployed, but also why the architecture was structured that way.

Week 4 – Sprint 4 Contributions

During Week 4, I continued building on the cloud deployment work, but with a stronger emphasis on performance testing, infrastructure expansion, operational practices, and adapting the project to a new use case. If Week 3 was about making the system cloud-deployable and structurally correct, Week 4 was more about extending that deployment knowledge into practical testing and system evolution.

A major Week 4 contribution was setting up an additional GPU EC2 deployment in Korea using a G6 instance. This followed the same infrastructure process as the earlier GPU EC2 setup, including configuring the VPC, subnets, Internet Gateway, route tables, and security groups, and launching the instance correctly with the right network design. The purpose of launching this Korea-based G6 instance was to test whether a more powerful GPU could significantly improve inference speed enough to better support near-real-time voice cloning, while also evaluating whether the additional cross-region latency would become a major issue. This required not just setup work, but also research into the trade-offs between region latency and GPU performance, because a faster GPU in a different region might still outperform a closer but weaker instance depending on the use case. That infrastructure experimentation was important because it linked deployment design directly to real-world performance concerns.

Another important Week 4 contribution was adopting more standard operational practices on AWS EC2, especially through the use of the ubuntu user and systemd-managed services. Rather than relying on manually starting processes, I configured systemd so that services such as the backend and inference-related components could be started, restarted, monitored, and persisted in a more production-oriented way. This also improved logging and observability through tools such as journalctl, service status inspection, and automatic restart behaviour. This matters because in a real deployment, reliability is not just about writing working code — it is also about how the services are managed, recovered, and monitored in the server environment.

In Week 4, I also deepened my understanding of the deployment structure and environment separation, especially around making the system work in both local mode and remote mode. This meant understanding how local development behaviour differs from the deployed AWS split-host setup, and ensuring that both could still be supported without breaking the architecture. That included understanding how the backend, GPU worker, GPT-SoVITS runtime, and storage paths interact differently depending on where the system is running. This stronger architectural understanding helped me ensure that the code and deployment flow were aligned instead of mixing local assumptions into the remote cloud design.

Another major Week 4 contribution was making code adjustments to support a new project direction after the supervisor changed the use case. Instead of only focusing on the original scenario from Weeks 1–2, which was mainly lecturer-oriented voice cloning, the system now also needed to support a speech-to-speech workflow where a clinician could have an assistant on standby in a consultation room. This meant the system had to evolve beyond simple text-to-speech generation and instead support a more live conversational pipeline.

To support this change, I reworked the Live page from a hold-to-speak plus browser transcript flow into a more controlled live speech pipeline using Faster Whisper. I removed browser speech recognition as the main transcription source because it was not accurate enough, and replaced it with a more reliable STT setup based on Faster Whisper. The live flow became: tap the microphone to start recording, detect pauses in speech, send phrase audio for transcription through Faster Whisper, take the transcribed text, send it into the existing TTS endpoint, and then show the generated clips for playback after the session ends. Importantly, I did not replace the main TTS pipeline itself; the existing cloned-voice generation endpoint remained in use, which meant I was extending the system rather than rewriting the entire cloning engine.

A substantial part of this speech-to-speech work involved improving speech detection and segmentation. The browser still listens to microphone input, but it now mainly acts as a phrase detector rather than deciding the actual transcript. In other words, the frontend handles simple voice activity and silence detection, while the actual words come from Faster Whisper. I also added a silence-before-inference slider to let the user control how long silence must last before a phrase is sent to STT. This made the live speech flow much more usable, since different speaking styles require different pause thresholds. If the silence window is too short, phrases get cut into tiny segments; if too long, responsiveness drops. Giving users control over this threshold was an important usability improvement in the live pipeline.

I also addressed an issue where Whisper could drift into Malay or Indonesian-looking transcripts when it was allowed to auto-detect language on short chunks. I fixed this by forcing the live ASR language to English by default, which improved consistency for the intended use case. In addition, generated audio clips were redesigned into a clearer live output structure, where each phrase becomes its own clip with states such as generating, ready, or error. Users can select clips from a list and download WAV files, making the live system more controllable and easier to inspect.

Another practical fix I implemented was disabling autoplay of generated clips during recording. This was important because if the system played generated audio while the microphone was still active, the microphone could pick up that generated sound and create feedback or contamination in the session. The new behaviour ensures that clips generate silently during recording, and only become playable after the session ends. I also added a final validation step where the full recording is sent through the existing full-audio transcription route after recording ends. This gives a more context-aware final transcript display without needing to regenerate all the already-created audio clips, improving transcript accuracy while keeping the live generation efficient.

On the backend side, I added the POST /api/live/transcribe-phrase endpoint to support the new live STT flow. In local mode, the frontend uploads phrase WAV files directly to the backend for transcription. In cloud and S3 mode, the frontend uploads phrase audio to S3, the backend passes the S3 reference to the GPU worker, and the GPU worker retrieves the file and runs transcription there. This was important because it ensured that the new speech-to-speech functionality remained compatible with the existing distributed AWS deployment, instead of becoming a feature that only works locally. I also noted the required cloud dependencies for this path, such as Faster Whisper, PyTorch, GPT-SoVITS runtime support, and the necessary S3 environment variables. Finally, I validated the implementation by confirming that the client build passed and that the affected server-side JavaScript and Python worker files were syntactically correct.

Overall Reflection (Weeks 3–4)

During Weeks 3 and 4, I learned a lot about how systems actually work outside of a local environment, especially when moving into cloud deployment. Earlier on, everything was running on my own machine, so things felt more straightforward. But once we moved to AWS, I realized that systems need to be designed very differently. Components like the backend, GPU worker, and GPT-SoVITS could no longer rely on being on the same machine, so I had to understand how to properly separate responsibilities and let each part communicate through APIs and shared storage like S3. This helped me better understand concepts like distributed systems and why clean architecture is important.

I also gained hands-on experience with AWS services such as EC2, VPC, subnets, and security groups. One thing that stood out to me was how small networking mistakes can completely break the system, even if the code is correct. For example, I learned that security groups are directional, and that both inbound and outbound rules matter. This experience improved how I troubleshoot problems, because I now think about both code and infrastructure instead of just focusing on one side.

Another thing I learned was how to think about performance more realistically. When testing different GPU instances and regions, I realized that having a stronger GPU does not always mean better performance if there is latency between regions. This helped me understand the trade-offs between compute power, network delay, and user experience, which is something I did not really consider before.

On the application side, I worked on improving how the system behaves, especially for long text and live speech. I learned that getting good output is not just about using a good AI model, but also about how you process inputs and handle outputs. Small changes like better chunking, smoother audio joining, and retry logic can make a big difference in quality. I also worked on the speech-to-speech feature, which made me understand how real-time systems need to handle timing, pauses, and user interaction more carefully compared to normal text-based systems.

Overall, these two weeks helped me move from just building features to understanding how to make a system work in a more realistic environment. I became more comfortable working with both code and cloud infrastructure, and I learned how to adapt the system when requirements change. It also made me more confident in handling unfamiliar problems and figuring things out step by step.
