# Week 7 - Session Contribution

During Week 7, I worked on improving the reliability and cost control of the cloud GPU deployment for the voice cloning system. I investigated an issue where the GPU EC2 instance stopped correctly when the idle timeout was set to one minute, but stayed running for much longer timeout values such as 30 or 45 minutes. I traced the issue through the Lambda idle-check flow, EventBridge schedule, GPU worker activity endpoint, frontend traffic, CloudFront routing, and the GPU worker request middleware. The investigation showed that passive requests from frontend pages, status checks, SSE progress routes, browser or CloudFront traffic, and possible test curl loops could refresh the GPU worker activity timer even when no real training or inference job was being used.

I updated the GPU worker activity tracking so the idle timer is no longer reset just because a route is called. Frontend restore calls, model list reads, inference status checks, SSE progress connections, health checks, and browser or CloudFront traffic no longer count as real user activity. The activity timer now refreshes only from real worker-side progress, such as active training subprocesses, active inference sessions, training logs, direct synthesis, model loading, and actual training or inference state transitions. This makes the idle shutdown behavior more accurate and prevents open pages or automated checks from keeping the GPU instance alive unnecessarily.

I also improved the Lambda idle shutdown logic to handle stuck training or inference states. If the GPU worker reports a busy-looking state but there has been no real recent activity for longer than the configured idle timeout, Lambda now treats it as stale busy state and can stop the EC2 instance instead of letting it run forever. I added regression tests for passive traffic, active worker detection, stale busy shutdown, and Lambda idle-check behavior, then verified both the GPU worker and Lambda test suites. I also updated the deployment documentation to explain how EventBridge, Lambda, and the GPU worker should interact for idle shutdown, including the fact that CloudFront/browser traffic and health checks should not reset the timer.

I also carried out detailed cloud debugging for the split deployment after the first rollout attempts failed. I traced the issue through the browser, CloudFront, Lambda, ALB listener rules, target groups, and the new inference worker service. At first, inference-related Lambda routes returned HTML error pages because only the browser SSE progress path had been routed to the inference worker, while model loading and inference status requests were still falling through to the default training worker target.
